Section 01

The Big Picture — How Java Works

☕ Layman's Take

Imagine you write a recipe in English. A translator converts it into a universal cooking language that any kitchen in the world can follow — regardless of whether the kitchen is in India, the US, or Japan. Java does exactly this. You write once, and it runs anywhere. That's Java's superpower: Write Once, Run Anywhere (WORA).

Every programming language needs to turn your human-readable code into instructions a computer can execute. Java does this in a two-step process that makes it uniquely portable:

1
Compilation: Your .java source file is compiled into .class bytecode — a platform-neutral intermediate format. Not binary for Windows, not binary for Mac. Just bytecode.
2
Execution: The JVM (Java Virtual Machine) reads that bytecode and translates it into native machine code specific to the host OS. The JVM is the bridge.

Section 02

JDK, JRE, JVM — The Layered Architecture

🎂 Layman's Take

Think of it as a Russian nesting doll. The smallest doll is the JVM. It sits inside the JRE (the middle doll). The JRE sits inside the JDK (the outermost doll). Each layer adds more capability on top of the one inside it.

Fig 2.1 — JDK ⊇ JRE ⊇ JVM — nested architecture (the "Russian doll")

JVM — The Heart

The JVM is a virtual machine — it simulates a real computer in software. Its job is to take platform-neutral bytecode and execute it on whatever OS it's running on. It also owns memory management completely — you never manually allocate or free memory in Java.

📖 Definition

The JVM (Java Virtual Machine) is a specification for a software-based computing machine that executes Java bytecode. It provides the runtime environment in which Java programs execute. Each OS has its own JVM implementation (HotSpot is Oracle's most popular one).

JRE — The Runtime

The JRE is what you install on a server or end-user machine just to run Java programs. It includes the JVM + all the standard library classes (java.lang, java.util, etc.). It does not include the compiler (javac).

JDK — The Developer's Toolkit

The JDK is what you install as a developer. It includes everything in the JRE, plus:

  • → javac — the compiler
  • → javadoc — documentation generator
  • → jar — packaging tool
  • → jdb — debugger
  • → jvisualvm — profiler
💡 Interview Tip

The most common interview trick: "I just need to run a Java program on a production server — do I install JDK or JRE?" Answer: JRE is enough (it has the JVM and runtime libs). JDK is only needed to develop/compile. In practice, Java 11+ ships JDK and JRE merged — but the conceptual distinction is still asked.

Section 03

How Your Code Executes — Step by Step

🎬 Layman's Take

Imagine producing a movie. You write a script (.java), a translator converts it into a storyboard (.class bytecode), and then the director (JVM) decides exactly how each scene will be filmed — adapting it to the city (OS) where filming is happening.

Fig 3.1 — Full Java execution pipeline from source to output

The JIT Compiler — Java's Secret Speed Weapon

The JVM starts by interpreting bytecode line by line (slower). As it runs, it identifies hot code — methods called frequently. The JIT (Just-In-Time) Compiler compiles those hot methods directly into native machine code. After the first few runs, those methods execute at near-native speed. This is why Java programs get faster the longer they run.

ℹ️ Fun Fact

Modern JVMs (like HotSpot) can be faster than C++ for long-running server processes because the JIT can make runtime optimizations (inline hot methods, eliminate dead branches) that a static compiler can't — it uses real runtime data to optimize.

Section 04

Heap vs Stack Memory — The Core of Every Java Interview

🏠 Layman's Take

Think of a restaurant. The Stack is the waiter's notepad — small, fast, used to remember the current order. When the order is done, the note is torn off and thrown away. The Heap is the kitchen storage — large, shared among all waiters, where the actual food (objects) are stored. The notepad only holds the locker number (reference) that points to where the food is in storage.

Fig 4.1 — Heap and Stack memory layout during program execution

Stack Memory

  • Stores method frames, local variables, and references to objects
  • LIFO — last in, first out. Frames are pushed/popped automatically
  • Each thread gets its own stack (thread-private)
  • Fixed size. StackOverflowError if it overflows (infinite recursion)
  • Very fast — just a pointer increment/decrement
  • Memory freed automatically when method returns

Heap Memory

  • Stores all objects created with new
  • Shared by all threads — requires synchronization for shared objects
  • Managed by Garbage Collector (GC)
  • Can grow/shrink dynamically. OutOfMemoryError if full
  • Slower than Stack — requires GC overhead
  • Objects live until GC decides to collect them

⚠️ Common Confusion

When you write Dog d = new Dog();, three things happen: (1) The variable d is created on the Stack, (2) The actual Dog object is created on the Heap, (3) The Stack variable d holds the memory address (reference) pointing to the Heap object. The variable and the object are in different places.

Method Area (Metaspace)

There's a third memory region often glossed over: the Method Area (called Metaspace in Java 8+). This is where the JVM stores:

  • → Class structure (bytecode, field names, method signatures)
  • → static variables and constants
  • → Runtime constant pool

Pre-Java 8, this was called PermGen and had a fixed size — leading to notorious OutOfMemoryError: PermGen space errors. Java 8+ replaced it with Metaspace, which grows dynamically from native memory.

Section 05

Garbage Collection — The JVM's Janitor

🗑️ Layman's Take

Imagine a hotel. Every guest is an object. When a guest checks out and no one else references their room number, the hotel staff (GC) eventually clears out the room and marks it available. You (the programmer) never explicitly clean up — the hotel staff handles it.

Garbage Collection is the process of automatically reclaiming heap memory occupied by objects that are no longer reachable from any live thread. Java uses generational GC based on the observation that most objects die young.

Fig 5.1 — Generational heap layout. Most objects die in Eden (Minor GC), survivors get promoted.

How GC Decides What to Collect

An object is eligible for GC when it is no longer reachable from any live thread through any chain of references. The GC uses reachability analysis starting from GC Roots:

  • → Local variables in active method frames (Stack)
  • → Static variables of loaded classes
  • → Active Java threads

Any object not reachable from a GC root = garbage. The GC marks and sweeps (or compacts) those objects.

💡 G1GC — Default in Java 9+

The G1 (Garbage First) GC is the default since Java 9. It divides the heap into equal-sized regions and prioritizes collecting regions with the most garbage first — giving more predictable pause times. For interview purposes: know that G1GC exists and that ZGC (Java 15+) achieves sub-millisecond pauses for very large heaps.

Section 06

Object Creation Lifecycle — Frame by Frame

Let's trace exactly what happens when you write Dog d = new Dog("Bruno");:

1
Class Loading: JVM checks if the Dog class is already loaded. If not, the ClassLoader loads it from the .class file into the Method Area.
2
Memory Allocation: JVM allocates space on the Heap for the new Dog object. The size equals the sum of all its instance fields (plus object header ~16 bytes).
3
Default Initialization: All instance fields get default values — int → 0, boolean → false, String → null, object refs → null.
4
Constructor Execution: The constructor Dog("Bruno") runs. Parent constructor runs first (implicit super() or explicit), then field initializers, then constructor body.
5
Reference Stored: The memory address of the new Dog object is stored in the variable d on the Stack.
Java — Annotated Example
public class Dog { // Instance variables → live on HEAP inside each Dog object String name; int age; // Constructor → runs on Heap when object is created public Dog(String name, int age) { this.name = name; // "this" refers to the current Heap object this.age = age; } public static void main(String[] args) { // Step 1: 'd' reference variable → created on STACK // Step 2: new Dog() → actual object created on HEAP at, say, 0x4A3C // Step 3: 'd' stores 0x4A3C (the address of the Heap object) Dog d = new Dog("Bruno", 3); // d.name → follow address 0x4A3C → find "Bruno" on Heap System.out.println(d.name); // Bruno System.out.println(d.age); // 3 // Local primitive → lives ENTIRELY on Stack, no Heap involvement int count = 5; // value 5 is directly on Stack // When main() ends → Stack frame is popped // 'd' variable disappears → Dog object at 0x4A3C has no more references // Dog object at 0x4A3C is now GC-eligible } }

⚠️ Classic Interview Trap

Primitives are copied by value. Object references are also copied by value — but the value is an address, so both variables point to the same object. Modifying through one affects the other. This is why Java is called "pass-by-value" — but for objects, the value is the reference.

Java

Section 08

Interview Questions & Model Answers

Top Interview Questions — Day 1 Topics

Q1: What is the difference between JDK, JRE, and JVM?
JVM is the execution engine that runs bytecode — it manages memory and provides runtime features. JRE = JVM + standard libraries — needed to run Java programs. JDK = JRE + developer tools — needed to write and compile Java. Nested: JVM ⊆ JRE ⊆ JDK. From Java 11+, JDK and JRE are merged, but the conceptual distinction remains important.
Q2: Where are objects stored in Java? Where are local variables?
Objects are stored on the Heap. Local variables and method call frames are stored on the Stack. When you write Dog d = new Dog(), the variable d (which holds the reference/address) is on the Stack, and the actual Dog object is on the Heap. Static variables go in the Method Area (Metaspace).
Q3: What triggers Garbage Collection? Can you force it?
GC runs automatically when the Eden space fills up (Minor GC) or when the JVM needs memory. You can hint to the JVM with System.gc(), but it's not guaranteed to run immediately — the JVM can ignore it. Never rely on explicit GC calls in production code. Objects become eligible when no live thread can reach them.
Q4: What is StackOverflowError vs OutOfMemoryError?
StackOverflowError: the call Stack is exhausted — typically caused by infinite recursion (a method calling itself with no base case). OutOfMemoryError: the Heap is full — too many live objects, or a memory leak (objects accumulating without being freed). Note: both are Errors, not Exceptions — they indicate JVM-level failures.
Q5: What is the String Pool? Why is it in the Heap?
The String Pool is a special cache inside the Heap (Metaspace before Java 8, Heap after) that stores string literals. When you write "Hello", the JVM first checks if "Hello" already exists in the pool — if yes, it reuses it rather than creating a new object. This saves memory since Strings are immutable and widely used. new String("Hello") bypasses the pool and creates a fresh Heap object.
Q6: Is Java pass-by-value or pass-by-reference?
Java is always pass-by-value. For primitives, the value itself is copied. For objects, the reference (address) is copied — so the method receives a copy of the pointer, not the object. If you reassign the reference inside the method, the caller's reference is unaffected. If you modify the object's fields through the reference, the caller sees the changes.
Q7: What is JIT compilation? How does it help?
JIT (Just-In-Time) compilation is the JVM's optimization mechanism. Initially, bytecode is interpreted line by line. The JVM identifies hot methods (called frequently) and compiles them to native machine code. After compilation, those methods run at near-native speed. This is why Java warm-up time matters in benchmarks — cold start = interpreter, warm = compiled native code.
