# Java Data Types — Complete Study Notes

---

## 1. Why Data Types Matter in Java

Three fundamental rules govern how Java handles data:

1. Every variable and every expression has a defined type.
2. Each data type is clearly and strictly defined by the language specification.
3. Every assignment is checked by the compiler for **type compatibility**.

Because of these rules, Java is classified as a **strongly typed programming language**. You cannot silently assign a `String` to an `int` or treat a `boolean` as a number — the compiler will catch it and refuse to compile.

> **Contrast with JavaScript:** In JS, `"5" + 3` silently produces `"53"`. In Java, this kind of implicit coercion between incompatible types is a compile-time error. That strictness is exactly what "strongly typed" means.

---

## 2. Is Java a Pure Object-Oriented Language?

**No.** Java is not considered a purely object-oriented language for two main reasons:

- Several OOP features are **not supported**, such as operator overloading and multiple inheritance of classes.
- Java depends on **primitive data types** (`int`, `char`, `boolean`, etc.), which are not objects. In a pure OOP language (like Smalltalk or Ruby), everything — including numbers and booleans — would be an object.

> **Interview Note:** Java bridges this gap partially through **wrapper classes** (`Integer`, `Character`, `Boolean`, etc.) and **autoboxing/unboxing** (introduced in Java 5), which allow primitives to behave like objects when needed. But at the memory and performance level, primitives remain non-objects.

---

## 3. Primitive Data Types Overview

Java has exactly **8 primitive data types**, divided into two families:

```
Primitive Data Types (8)
├── Numeric
│   ├── Integral ──── byte, short, int, long
│   └── Floating-Point ── float, double
└── Non-Numeric
    ├── char
    └── boolean
```

**Signed vs. Unsigned:**
All primitive data types except `boolean` and `char` are **signed**, meaning they can represent both positive and negative values. The `char` type is **unsigned** (range 0 to 65535), and `boolean` has no numeric range at all.

## 4. Integral Data Types

### 4.1 byte

**Size:** 1 byte (8 bits)
**Range:** -128 to +127
**Wrapper Class:** `Byte`
**Default Value:** `0`

**How the 8 bits are laid out:**

```
[Sign] [2^6] [2^5] [2^4] [2^3] [2^2] [2^1] [2^0]
  1 bit                   7 data bits
```

- The **most significant bit (MSB)** is the sign bit: `0` = positive, `1` = negative.
- Positive numbers are stored directly in binary.
- Negative numbers are stored in **two's complement** form.

**Two's complement in simple terms:** To find the two's complement of a number, flip all the bits (0 becomes 1, 1 becomes 0) and then add 1. This is how the JVM represents negative values internally. For example, `-5` in a byte is stored as `11111011`.

**Assignment examples:**

```java
byte b = 10;        // valid
byte b = 127;       // valid — maximum value
byte b = 128;       // CE: possible loss of precision, found: int, required: byte
byte b = true;      // CE: incompatible types, found: boolean, required: byte
byte b = "shreyas"; // CE: incompatible types, found: String, required: byte
```

**Best use case:** `byte` is the natural choice when handling data in the form of **streams** — either from a file or over a network. File I/O and network I/O in Java operate on byte streams (`InputStream`, `OutputStream`), making `byte` and `byte[]` the fundamental unit of raw data transfer.

---

### 4.2 short

**Size:** 2 bytes (16 bits)
**Range:** -2^15 to 2^15 - 1 → **-32,768 to 32,767**
**Wrapper Class:** `Short`
**Default Value:** `0`

**Assignment examples:**

```java
short s = 32767;   // valid — maximum value
short s = 32768;   // CE: possible loss of precision, found: int, required: short
short s = 10.5;    // CE: possible loss of precision, found: double, required: short
short s = true;    // CE: incompatible types, found: boolean, required: short
```

**Best use case:** The `short` type was designed for **16-bit processors** (like the Intel 8085). Since modern processors are 32-bit or 64-bit, `short` is rarely used today. It is essentially an **outdated data type** — you will almost always use `int` instead. The only practical scenario is when memory savings matter in very large arrays.

---

### 4.3 int

**Size:** 4 bytes (32 bits)
**Range:** -2^31 to 2^31 - 1 → **-2,147,483,648 to 2,147,483,647** (roughly ±2.1 billion)
**Wrapper Class:** `Integer` (note: **not** `Int`)
**Default Value:** `0`

This is the **most commonly used** data type for whole numbers in Java.

**Assignment examples:**

```java
int x = 2147483647;    // valid — maximum value
int x = 2147483648;    // CE: integer number too large
int x = 2147483648L;   // CE: possible loss of precision, found: long, required: int
int x = true;          // CE: incompatible types, found: boolean, required: int
```

> **Why does `2147483648` fail even without assignment?** Because integer literals in Java are `int` by default. The value 2,147,483,648 exceeds the `int` range, so the compiler rejects the literal itself before it even considers the assignment. Appending `L` makes it a `long` literal, but then you cannot assign a `long` to an `int` without explicit casting.

---

### 4.4 long

**Size:** 8 bytes (64 bits)
**Range:** -2^63 to 2^63 - 1 (approximately ±9.2 × 10^18)
**Wrapper Class:** `Long`
**Default Value:** `0L` (or simply `0`)

Use `long` when `int` is not large enough. Common real-world scenarios:

- The distance light travels in a year (in meters) exceeds `int` range.
- The number of characters in a very large file — this is why `File.length()` returns `long`, not `int`.
- Timestamps in milliseconds since epoch (`System.currentTimeMillis()` returns `long`).

> **Literal suffix:** To write a `long` literal, append `L` or `l` to the number: `long x = 9876543210L;`. Lowercase `l` is legal but discouraged because it looks like the digit `1`.

---

### Summary of All Integral Types

| Type | Size | Range | Default | Wrapper Class |
|---|---|---|---|---|
| `byte` | 1 byte | -128 to 127 | `0` | `Byte` |
| `short` | 2 bytes | -32,768 to 32,767 | `0` | `Short` |
| `int` | 4 bytes | -2,147,483,648 to 2,147,483,647 | `0` | `Integer` |
| `long` | 8 bytes | -2^63 to 2^63 - 1 | `0` | `Long` |

**Note:** All four integral types can only represent whole numbers. For decimal values, you need floating-point types.

## 5. Floating-Point Data Types

### 5.1 float

**Size:** 4 bytes (32 bits)
**Precision:** ~5 to 6 decimal places of accuracy
**Standard:** IEEE 754 Single Precision
**Range:** approximately ±3.4 × 10^38
**Wrapper Class:** `Float`
**Default Value:** `0.0f`

```java
float f = 10.5f;    // valid — the 'f' suffix is mandatory
float f = 10.5;     // CE: possible loss of precision, found: double, required: float
```

> **Why is the `f` suffix required?** All decimal literals in Java default to `double`. Writing `10.5` creates a `double` value. Since `double` is wider than `float`, assigning it to a `float` variable causes a potential loss of precision, and the compiler blocks it.

---

### 5.2 double

**Size:** 8 bytes (64 bits)
**Precision:** ~14 to 15 decimal places of accuracy
**Standard:** IEEE 754 Double Precision
**Range:** approximately ±1.7 × 10^308
**Wrapper Class:** `Double`
**Default Value:** `0.0`

```java
double d = 10.5;     // valid — decimal literals are double by default
double d = 10.5d;    // also valid — explicit suffix, but optional
```

**When to use `float` vs. `double`:** In almost all cases, prefer `double`. It is the default for decimal literals and provides significantly better precision. Use `float` only when memory is critical (e.g., large arrays in graphics or scientific computing) and you can tolerate the reduced precision.

> **Neither `float` nor `double` should be used for monetary/currency calculations** — they introduce rounding errors due to how IEEE 754 represents fractions in binary. Use `BigDecimal` for financial arithmetic.

---

### Floating-Point Summary

| Type | Size | Precision | Range | Default | Wrapper |
|---|---|---|---|---|---|
| `float` | 4 bytes | ~5-6 decimal digits | ±3.4 × 10^38 | `0.0f` | `Float` |
| `double` | 8 bytes | ~14-15 decimal digits | ±1.7 × 10^308 | `0.0` | `Double` |

## 6. boolean Data Type

**Size:** Not precisely defined by the JVM specification (implementation-dependent, but logically 1 bit of information)
**Range:** Not applicable — only two allowed values: `true` and `false`
**Wrapper Class:** `Boolean`
**Default Value:** `false`

**Assignment examples:**

```java
boolean b = true;     // valid
boolean b = 0;        // CE: incompatible types, found: int, required: boolean
boolean b = True;     // CE: cannot find symbol — Java is case-sensitive, 'True' is not a keyword
boolean b = "True";   // CE: incompatible types, found: String, required: boolean
```

**Critical difference from C/C++:** In C and C++, integers can serve as booleans (`0` is false, non-zero is true). Java does **not** allow this. You cannot use an `int` where a `boolean` is expected:

```java
int x = 0;
if (x) {                   // CE: incompatible types, found: int, required: boolean
    System.out.println("hello");
}

while (1) {                 // CE: incompatible types, found: int, required: boolean
    System.out.println("hello");
}
```

The correct Java equivalents would be `if (x != 0)` and `while (true)`.

## 7. char Data Type

**Size:** 2 bytes (16 bits)
**Range:** 0 to 65,535 (unsigned)
**Wrapper Class:** `Character` (note: **not** `Char`)
**Default Value:** `'\u0000'` (the null character, which represents a blank/space when printed)

**Why is `char` 2 bytes in Java?**

Older languages like C use the **ASCII** encoding system, which defines only 128 characters (or 256 in extended ASCII). Since all ASCII characters fit within 8 bits, `char` in C is just 1 byte.

Java, however, uses the **Unicode** encoding system, which supports characters from virtually every written language in the world — Latin, Cyrillic, Chinese, Arabic, Devanagari, emojis, and more. The original Unicode standard required 16 bits to represent all characters, so Java's `char` was designed as **2 bytes**.

> **Modern note:** Unicode has since expanded beyond 65,536 characters (the Basic Multilingual Plane). Characters outside this range (like many emojis and rare scripts) require **surrogate pairs** — two `char` values to represent a single character. This is why `String.length()` can sometimes return a number larger than the actual number of visible characters.

```java
char c = 'A';        // valid — character literal
char c = 65;         // valid — assigns the character with Unicode value 65, which is 'A'
char c = '\u0041';   // valid — Unicode escape for 'A'
```

## 8. Complete Primitive Data Types Reference

| Type | Size | Range | Default | Wrapper Class | Signed? |
|---|---|---|---|---|---|
| `byte` | 1 byte | -128 to 127 | `0` | `Byte` | Yes |
| `short` | 2 bytes | -32,768 to 32,767 | `0` | `Short` | Yes |
| `int` | 4 bytes | ±2.1 billion | `0` | `Integer` | Yes |
| `long` | 8 bytes | ±9.2 × 10^18 | `0` | `Long` | Yes |
| `float` | 4 bytes | ±3.4 × 10^38 | `0.0f` | `Float` | Yes |
| `double` | 8 bytes | ±1.7 × 10^308 | `0.0` | `Double` | Yes |
| `char` | 2 bytes | 0 to 65,535 | `'\u0000'` | `Character` | No |
| `boolean` | JVM-dependent | true / false | `false` | `Boolean` | N/A |

**Wrapper class spelling traps for exams:**
- `int` wraps to `Integer` (not `Int`)
- `char` wraps to `Character` (not `Char`)
- All other wrappers match the primitive name with a capital first letter.
